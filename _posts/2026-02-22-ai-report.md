#### Claude Code v2.1.50: Opus 4.6 ファストモード 1M コンテキスト対応・Worktree フックイベント API 追加
v2.1.50（2026年2月21日リリース）にて Opus 4.6 のファストモードが1Mコンテキストウィンドウに対応した。エージェント定義で `isolation: worktree` の宣言的指定が可能になり、`WorktreeCreate`・`WorktreeRemove` フックイベントが追加されてカスタムVCSのセットアップ/解体が自動化できるようになった。`CLAUDE_CODE_DISABLE_1M_CONTEXT` 環境変数が追加され無効化も可能。長時間セッション向けのメモリリーク修正も多数含まれる。

### 2. GitHub Copilot

#### Gemini 3.1 Pro が GitHub Copilot でパブリックプレビュー開始
GoogleのGemini 3.1 Proが2026年2月19日よりGitHub Copilotに追加。本モデルはedit-then-testループの効率化とツール精度の高さを特長とし、少ないツール呼び出しで高い解決率を達成する。VS Code・Visual Studio・github.com・GitHub Mobileのchat/ask/edit/agentモードすべてで利用可能。Copilot Pro/Pro+/Business/Enterpriseユーザー向けに段階的にロールアウト中。
> ソース: [github.blog](https://github.blog/changelog/2026-02-19-gemini-3-1-pro-is-now-in-public-preview-in-github-copilot)

#### Copilot コーディングエージェントのモデルピッカーが Business・Enterprise ユーザーに開放
非同期・自律型バックグラウンドエージェント「Copilot coding agent」のモデル選択機能がCopilotビジネスおよびEnterpriseユーザーに拡張された。選択可能なモデルはClaude Opus 4.5/4.6、Claude Sonnet 4.5/4.6、GPT-5.1-Codex-Max、GPT-5.2-Codex、GPT-5.3-Codexなど。Autoモードではパフォーマンスと可用性を元にモデルを自動最適化する。
> ソース: [github.blog](https://github.blog/changelog/2026-02-19-model-picker-for-copilot-coding-agent-for-copilot-business-and-enterprise-users)

#### 組織レベルの Copilot 利用メトリクスダッシュボードがパブリックプレビューで公開
従来はEnterprise管理者のみが参照できたCopilot利用状況ダッシュボードが、組織（Organization）単位でも閲覧可能になった。組織オーナーまたは「View Organization Copilot Metrics」カスタムロール保有者が対象で、EnterpriseおよびスタンドアロンのFree/Teamプランも利用可。採用率・利用傾向の組織スコープ分析が可能となり、Enterprise全体への権限委譲なしに細かい利用状況把握が実現する。
> ソース: [github.blog](https://github.blog/changelog/2026-02-20-organization-level-copilot-usage-metrics-dashboard-available-in-public-preview)

#### GitHub Copilot の Zed エディタ対応が一般提供（GA）開始
RustベースのハイパフォーマンスエディタZedとGitHub Copilotの正式パートナーシップにより、既存のCopilot Pro/Pro+/Business/Enterpriseサブスクリプションでそのまま利用可能に。ZedのSettings→「GitHub Copilot Chat」セクションからサインインするだけで設定完了。追加のAIライセンス不要でCopilotをZed環境にシームレスに統合できる。
> ソース: [github.blog](https://github.blog/changelog/2026-02-19-github-copilot-support-in-zed-generally-available)

#### Copilot 利用メトリクス API にプルリクエストスループットとマージ時間指標が追加
GitHub CopilotメトリクスAPIに、PRレビュー提案数・承認数・Copilotコーディングエージェント経由でマージされたPR数・PRサイクルタイム（マージまでの所要時間）の指標が新たに追加。Copilot利用メトリクスを有効化したEnterpriseのAdminまたは「View Enterprise Copilot Metrics」ロール保有者が対象。Copilotが開発サイクル全体に与える定量的インパクトを計測可能にする。
> ソース: [github.blog](https://github.blog/changelog/2026-02-19-pull-request-throughput-and-time-to-merge-available-in-copilot-usage-metrics-api)

2026年2月17日付で、GitHub CopilotのすべてのUIからClaude Opus 4.1（→ Claude Opus 4.6 推奨）、GPT-5（→ GPT-5.2 推奨）、GPT-5-Codex（→ GPT-5.2-Codex 推奨）が廃止された。Enterpriseでは管理者がモデルポリシーを更新する必要がある場合あり。廃止済みモデルの手動削除作業は不要。

#### AI が開発者の技術選択を再形成している — Octoverse 2025 データ分析
GitHub Copilotを早期に利用した開発者の行動データをもとに、AIの「コンビニエンスループ」が技術選択を変化させているとの分析記事。新規開発者の80%が最初の1週間以内にCopilotを使用し、型付き言語（TypeScript）がAI補完の恩恵を受けて急成長（前年比+66%）したことが示される。開発組織向けのアーキテクチャ管理とAIメトリクス活用の実践的指針も提示。

### 4. AIコーディングエージェント比較・ベンチマーク

#### コンテキストファイルはコーディングエージェントの性能を悪化させる可能性 — ETH Zurich 研究
ETH Zurichの研究者がClaude Code・Codex・Qwen Codeを含む4つのコーディングエージェントをSWE-bench Liteおよびカスタムベンチマーク（138タスク）で評価。LLMが自動生成したコンテキストファイル（AGENTS.md等）は8つのテスト設定のうち5つで成功率を低下させ、推論コストを20%以上増加させた。人間が手書きしたコンテキストファイルは平均約4ポイントの改善に留まり、効果は限定的と結論付けられた。
> ソース: [the-decoder.com](https://the-decoder.com/context-files-for-coding-agents-often-dont-help-and-may-even-hurt-performance/)

#### Gemini 3.1 Pro Preview、Artificial Analysis Intelligence Index でトップ — コスト半額以下
Artificial Analysisの統合AIベンチマーク（10項目）でGemini 3.1 Pro Previewが57点を獲得し、Claude Opus 4.6（53点）やGPT-5.2（51点）を上回った。エージェントベースコーディング・知識・科学的推論を含む6カテゴリで1位を記録しつつ、フルベンチマーク実行コストは$892とGPT-5.2の$2,304・Claude Opus 4.6の$2,486の半額以下。ただし実世界のエージェントタスクではClaude Sonnet 4.6・Opus 4.6・GPT-5.2に後れを取る点も課題として残っている。
> ソース: [the-decoder.com](https://the-decoder.com/googles-gemini-3-1-pro-preview-tops-artificial-analysis-intelligence-index-at-less-than-half-the-cost-of-its-rivals/)

#### AWS AI コーディングツール「Kiro」が本番環境を「削除・再作成」— 13時間の障害を引き起こしたと報道
Financial Timesの報道によると、AWS自社のAIコーディングツールKiroが関与した本番障害が少なくとも2件発生。うち1件はKiroが顧客向けシステムの「削除と再作成」を実行し13時間の障害を招いた。AWSはユーザーエラー（アクセス制御の設定ミス）が原因と否定しているが、当時AIツールにはオペレーターレベルの権限が付与されており、同僚レビューなしに操作が行われていたことが判明した。
> ソース: [the-decoder.com](https://the-decoder.com/aws-ai-coding-tool-decided-to-delete-and-recreate-a-customer-facing-system-causing-13-hour-outage-report-says/)

#### Anthropic、Claude Code にデスクトップ機能を追加 — 開発サーバー起動・PR 自動レビュー・CI 自動修正など
AnthropicはClaude Codeに新たなデスクトップ機能を展開し、開発サーバーの起動・実行中WebアプリのUI内表示・エラーの自動検出と修正が可能になった。GitHubプロジェクトではPRをバックグラウンドで監視し、CIエラーを自動修正してテスト通過後にPRをマージする機能も追加。CLI・デスクトップ・Web・モバイル間でセッションが継続されるクロスプラットフォーム対応も実現した。
> ソース: [the-decoder.com](https://the-decoder.com/anthropic-updates-claude-code-with-desktop-features-that-automate-more-of-the-dev-workflow/)

### 5. AI開発ワークフロー・ベストプラクティス

#### 4つの AI コーディングツールのコンテキストウィンドウ比較分析（3,177 API コール傍受）
Claude Opus 4.6・Sonnet 4.5・GPT-5.3 Codex・Gemini 2.5 Proの4ツールに同一バグ修正タスクを与え、コンテキスト構成を比較。OpusはAPIコールの95%がキャッシュヒットで23Kトークンと最も効率的だが16.4Kのツール定義オーバーヘッドがあり、Geminiはgitログ全体（118.5Kトークン）を読み込む非効率な戦略を取り平均258Kトークンを消費した。各ツールのコンテキスト戦略の違いが明確に可視化され、開発フローのコスト最適化に直接役立つ実証データ。
> ソース: [theredbeard.io](https://theredbeard.io/blog/i-intercepted-3177-api-calls-across-4-ai-coding-tools)

#### OpenClaw スキル定義のセキュリティリスク分析：CLAUDE.md・.cursorrules 汚染攻撃を含む 15 カテゴリ
AIエージェントのスキル定義を静的解析するスキャナーを構築し、ClawHubで発見された15カテゴリのリスクパターンを報告。特に注目は「メモリ/設定汚染」カテゴリで、CLAUDE.md・SOUL.md・.cursorrules・.windsurfrules等への書き込みを指示することでセッション間に永続する悪意ある命令を埋め込む攻撃が確認された。プロンプトインジェクション・シェル実行・データ窃取・埋め込みシークレットなども対象で、OWASP LLM Top 10参照付き。
> ソース: [safeclaw.io](https://safeclaw.io/)

#### Rigour v4 — AI コーディングエージェント向けオープンソース品質ゲート CLI
AIエージェントが書いたコードに対して「AST→LLM→AST検証」の3段階ディープ分析を実施。ASTで決定論的なファクト（行数・ネスト深度・メソッドシグネチャ）を抽出し、LLMがSOLID違反やGodクラスを解釈、最後にASTでLLMのハルシネーションを検証することでゼロハルシネーションを実現（88/88 AST検証済み）。エージェントがコード品質を劣化させないためのワークフロー組み込みを想定した設計。
> ソース: [rigour.run](https://rigour.run)

#### Prothon — AI エージェント向けドキュメント駆動型 Python プロジェクトジェネレーター
セッション間でのコンテキストドリフト問題に対し、SPEC/DESIGN/PATTERNSの3層ドキュメント階層でAIの実装根拠を管理するアプローチを提案。タスク実行前にエージェントが「変更予定ファイル・成功基準・行数予測」をpromiseとして宣言し、実際のgit diffと照合することでドリフトを定量化する仕組みを実装。ruff/ty/pytest/hypothesisなど8つの品質ツールを同梱し、Claude Code向けに最適化されているがエージェント非依存設計。
> ソース: [prothon.dev](https://www.prothon.dev/)

#### AstrMap — RAG を使わない AI エージェント向け AST「マップ」生成ツール
コードのベクトル検索（RAG）に依存せず、リポジトリをミリ秒で解析してAI読み取り可能なAST圧縮マップ（行番号・関数付き目次）を生成するGoツール。LLMはこのマップをほぼゼロトークンコストで受け取り、修正が必要なファイルを確定的に特定できるため、エージェントのコンテキスト肥大化を防ぐ。Go/JS/TS/Python/HTML/CSSに対応し、ローカル実行の単一バイナリとして提供。
> ソース: [github.com](https://github.com/hubby247/astrmap)

#### Corral — AI コーディングエージェントが設定できる認証・Stripe 課金スキャフォールダー
AIエージェントが「認証と決済を追加して」という指示でハルシネーションする問題に対し、llms.txt形式の機械可読仕様を提供するCLIツール。Express/Next.js/Hono/Fastifyなど13以上のフレームワークを検出し既存サーバーに組み込む設計で、3つのAIモデル×3ラウンドのブラインドエージェントテストで検証済み。全コマンドが--jsonをサポートし、エラー出力に「fix」フィールドを含むエージェント統合を前提とした設計。
> ソース: [github.com](https://github.com/llama-farm/corral)

#### 関数型プログラミング（Elixir）はなぜ AI コーディングエージェントに最適か
FPの純粋関数・不変性・副作用の境界集約がAIエージェントのトークン効率・パターン認識・ベストプラクティス強制の3面で有利であるという実践的考察。LLMは純粋関数を非常に得意とし、副作用をエッジに集約するFPアーキテクチャにより人間とAIの役割分担が明確化される。OOP的副作用の散在はモデルのコンテキスト肥大化と認知負荷増大を招くと指摘。
> ソース: [reddit.com](https://www.reddit.com/r/programming/comments/1r8nbtz/)

#### Alive-analysis — AI エージェント向け Git トラッキング分析ワークフローキット
AIコーディングエージェントとの分析作業が「使い捨てのチャット」になってしまう問題を解決するオープンソースツール。ASK→LOOK→INVESTIGATE→VOICE→EVOLVEの5段階ループとチェックリストを強制し、分析結果をGitトラッキングのMarkdownファイルとして保存することで再現性と追跡可能性を確保。Claude CodeとCursorで動作し、クイックモード（1ファイル）とフルモード（5ファイル）を選択可能。
> ソース: [github.com](https://github.com/with-geun/alive-analysis)
